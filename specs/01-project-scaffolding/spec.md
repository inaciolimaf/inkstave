# Spec 01 — Project Scaffolding (requirements)

## 1. Summary

This spec establishes the Inkstave monorepo: the top-level directory layout,
Python tooling (`uv`), Node tooling (`pnpm`), a base `docker-compose.yml` that
runs only the infrastructure services needed early (PostgreSQL + Redis), a
documented `.env.example`, an MIT `LICENSE`, a `justfile` command runner, and
pre-commit hooks. It contains **no application code**; its job is to give every
later spec a well-defined, conventional place to add code and a reproducible
local environment.

## 2. Context & dependencies

- **Depends on:** none. This is the first spec.
- **Unlocks:** spec 02 (backend app needs `backend/` + `uv` project), spec 03
  (DB needs the Postgres compose service + env vars), spec 04 (CI references the
  layout and command runner), and every spec thereafter.
- **Affected areas:** repository root, `backend/`, `frontend/`, `collab/`,
  `infra/`, `docs/`.

## 3. Goals

- Create the canonical top-level directory layout matching `README.md`.
- Establish Python tooling with **`uv`**: a `backend/pyproject.toml` declaring
  the project, `ruff` (lint + format) config, and a static type-checker
  (`mypy`) config — pinned Python `>=3.12`.
- Establish Node tooling with **`pnpm`**: a `frontend/package.json` placeholder
  and root `pnpm-workspace.yaml`; Vite is *declared as the build tool to be
  configured later* but no React app is created here.
- Provide a base `docker-compose.yml` at the repo root with **only** `postgres`
  and `redis` services (named volumes, healthchecks, env-driven config). App
  services (`backend`, `frontend`, `worker`) are added by later specs.
- Provide an `.env.example` documenting every variable used at this stage, with
  safe non-secret defaults.
- Add an MIT `LICENSE` (Inkstave is MIT; Overleaf code is never copied).
- Add a `justfile` (command runner) with common commands.
- Add `.pre-commit-config.yaml` running `ruff` + basic hygiene hooks.
- Add a root `.gitignore` and `.editorconfig`.
- Add `docs/adr/0001-tooling-choices.md` recording the `uv`/`pnpm`/`just`
  decisions.

## 4. Non-goals (explicitly out of scope)

- Any FastAPI application code, routers, settings module (spec 02).
- Any database engine, models, Alembic config or migrations (spec 03).
- Any pytest/Vitest/Playwright config or test code (spec 04).
- Any React/Vite frontend source (spec 09).
- `backend`/`frontend`/`worker` services in compose, Dockerfiles for the app,
  nginx, or production packaging (later specs, e.g. 56).
- CI pipeline definition (spec 04 owns the GitHub Actions workflow).

## 5. Detailed requirements

### 5.1 Data model (if any)

None. This spec introduces no persisted data and no schema. The Postgres
service is started empty; spec 03 owns the schema.

### 5.2 Backend / API (if any)

No API. Backend deliverable is the **project skeleton only**:

```
backend/
├── pyproject.toml          # project metadata + deps groups + tool config
├── README.md               # one-paragraph "this is the FastAPI backend" stub
├── src/
│   └── inkstave/
│       └── __init__.py     # empty package marker, __version__ = "0.0.0"
└── tests/
    └── __init__.py
```

`backend/pyproject.toml` requirements:

- `[project]`: `name = "inkstave-backend"`, `version = "0.0.0"`,
  `requires-python = ">=3.12"`. No runtime dependencies are added here yet
  (FastAPI etc. arrive in spec 02); only declare the dependency *groups*
  structure so later specs append to it.
- Use **`uv`** as the package/dependency manager. Include a `[tool.uv]` section
  and commit a `uv.lock` (generated by `uv lock`, even if dependency-light).
- `[dependency-groups]` (PEP 735) with empty-but-present `dev` group, ready for
  spec 02/04 to add `ruff`, `mypy`, `pytest`, etc. It is acceptable to add
  `ruff` and `mypy` to `dev` now since this spec configures them.
- `[tool.ruff]`: `line-length = 100`, `target-version = "py312"`; enable rule
  sets `E`, `F`, `I` (isort), `UP`, `B`, `ASYNC`; `[tool.ruff.format]` with
  defaults. Exclude `migrations/` (forward-looking).
- `[tool.mypy]`: `python_version = "3.12"`, `strict = true`,
  `warn_unused_ignores = true`, `plugins = []` (Pydantic/SQLAlchemy plugins
  added by later specs). Scope to `src/`.

### 5.3 Frontend / UI (if any)

No UI. Frontend deliverable is the **skeleton only**:

frontend/
├── package.json            # name + private:true; "type":"module"; no app deps
├── README.md               # stub: "Vite/React/TS app, scaffolded in spec 09"

### 5.5 Configuration

#### Root `docker-compose.yml`

Define exactly two services plus named volumes. No `version:` key (Compose v2).

- **`postgres`**
  - `image: postgres:16-alpine`
  - `environment`: `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB` taken
    from the env file (`${POSTGRES_USER}`, etc.).
  - `ports`: `"${POSTGRES_PORT:-5432}:5432"`.
  - `volumes`: `pgdata:/var/lib/postgresql/data`.
  - `healthcheck`: `pg_isready -U ${POSTGRES_USER}`, interval `5s`, retries `5`.
- **`redis`**
  - `image: redis:7-alpine`
  - `command: ["redis-server", "--save", "", "--appendonly", "no"]`
    (ephemeral by default; persistence is not needed yet).
  - `ports`: `"${REDIS_PORT:-6379}:6379"`.
  - `healthcheck`: `redis-cli ping` expecting `PONG`.
- `volumes:` declares `pgdata:`.
- `env_file: .env` so values come from the developer's `.env`.

#### `.env.example`

A committed template (never `.env` itself; `.env` is git-ignored). Document
every variable with a comment. At this stage:

| Variable | Example / default | Purpose |
| --- | --- | --- |
| `POSTGRES_USER` | `inkstave` | Postgres role |
| `POSTGRES_PASSWORD` | `inkstave` | Postgres password (dev only) |
| `POSTGRES_DB` | `inkstave` | Postgres database name |
| `POSTGRES_PORT` | `5432` | host port mapping |
| `REDIS_PORT` | `6379` | host port mapping |
| `DATABASE_URL` | `postgresql+asyncpg://inkstave:inkstave@localhost:5432/inkstave` | full async DSN consumed by spec 02/03 |
| `REDIS_URL` | `redis://localhost:6379/0` | consumed by spec 02 |

> `DATABASE_URL`/`REDIS_URL` are added now (even though no code reads them yet)
> so spec 02 inherits a complete `.env.example`. Later specs append rows; they
> never remove these.

#### `justfile`

A `justfile` at the repo root with at least these recipes (names exact):

- `just up` → `docker compose up -d` (starts postgres + redis).
- `just down` → `docker compose down`.
- `just logs` → `docker compose logs -f`.
- `just fmt` → `uv run ruff format backend` (and a placeholder note for
  frontend prettier, added in spec 09).
- `just lint` → `uv run ruff check backend && uv run mypy backend/src`.
- `just bootstrap` → `cp -n .env.example .env || true` then
  `uv sync --project backend` and `pnpm install` (frontend install is a no-op
  until spec 09 adds deps; the recipe must not fail if there is nothing to do).
- `just hooks` → `uv run pre-commit install`.

Recipes that call tooling not yet installed (e.g. frontend test) are **not**
added in this spec.

#### `.pre-commit-config.yaml`

Hooks:

- `ruff` (lint, `--fix`) and `ruff-format` from `astral-sh/ruff-pre-commit`.
- `trailing-whitespace`, `end-of-file-fixer`, `check-yaml`,
  `check-added-large-files`, `check-merge-conflict` from `pre-commit-hooks`.

Pin each hook repo to a concrete release tag (no floating refs).

#### `.gitignore` (root)

Ignore at least: `.env`, `__pycache__/`, `*.pyc`, `.venv/`, `.mypy_cache/`,
`.ruff_cache/`, `node_modules/`, `dist/`, `build/`, `.pnpm-store/`,
`*.log`, `.pytest_cache/`, `htmlcov/`, `.coverage`, OS files
(`.DS_Store`). **Do not** ignore `uv.lock` or `pnpm-lock.yaml` (lockfiles are
committed).

#### `.editorconfig` (root)

UTF-8, LF line endings, final newline, trim trailing whitespace; 4-space indent
for Python, 2-space for JS/TS/JSON/YAML/Markdown.

## 6. Overleaf reference (study only — never copy)

> Read these in `../overleaf/` to understand monorepo orchestration and dev
> ergonomics only. Inkstave uses a *different* layout (single backend service,
> not many micro-services) and a different stack (Python/uv vs Node/yarn). Take
> ideas, not code.

- `package.json` (repo root) — verified present. Overleaf uses Yarn workspaces
  to manage many packages; study how a monorepo declares workspaces. Inkstave
  uses `pnpm-workspace.yaml` for the (single) frontend and `uv` for the backend
  instead.
- `docker-compose.yml` (repo root) — verified present. Study how services,
  named volumes, and env wiring are composed. Inkstave's base compose is far
  smaller (only postgres + redis at this stage).
- `develop/` (`develop/docker-compose.yml`, `develop/docker-compose.dev.yml`,
  `develop/dev.env`, `develop/README.md`) — all verified present. Study how a
  developer brings the stack up locally and how a `dev.env` documents
  configuration. Inkstave mirrors the *idea* with `.env.example` + `justfile`.
- `server-ce/` (layout: `bin/`, `config/`, `nginx/`, `Makefile`,
  `services.js`) — verified present. Study how the community-edition packaging
  is organised. Inkstave keeps an `infra/` directory for the equivalent assets
  (most arrive in spec 56).

No Overleaf equivalent exists for: a `uv`-based Python monorepo, a `justfile`
runner, or `pnpm-workspace.yaml` — Overleaf is Node/Yarn + Makefiles. Design
those independently from the tooling docs.

## 7. Acceptance criteria

1. **Given** a fresh clone, **when** I run `ls`, **then** the directories
   `backend/`, `frontend/`, `collab/`, `infra/`, `docs/`, `specs/` and the
   files `docker-compose.yml`, `.env.example`, `LICENSE`, `justfile`,
   `.pre-commit-config.yaml`, `.gitignore`, `.editorconfig`,
   `pnpm-workspace.yaml` all exist.
2. **Given** the repo, **when** I open `LICENSE`, **then** it is the MIT License
   text with the correct copyright holder line, and it contains no AGPL text.
3. **Given** the repo, **when** I run `cp .env.example .env && just up`, **then**
   `docker compose ps` shows `postgres` and `redis` both `healthy` within 30s.
4. **Given** the running stack, **when** I run
   `docker compose exec postgres pg_isready -U "$POSTGRES_USER"`, **then** it
   reports the server accepting connections; **and** `docker compose exec redis
   redis-cli ping` returns `PONG`.
5. **Given** `backend/pyproject.toml`, **when** I run `uv sync --project
   backend`, **then** it resolves and writes/uses `backend/uv.lock` with no
   error.
6. **Given** the backend skeleton, **when** I run `uv run ruff check backend`
   and `uv run ruff format --check backend`, **then** both pass with no
   findings.
7. **Given** `backend/src/inkstave/__init__.py`, **when** I run
   `uv run mypy backend/src`, **then** it reports success (no type errors).
8. **Given** `.pre-commit-config.yaml`, **when** I run `pre-commit run
   --all-files`, **then** every hook passes (after any auto-fixes are
   committed).
9. **Given** the `justfile`, **when** I run `just --list`, **then** it lists at
   least: `up`, `down`, `logs`, `fmt`, `lint`, `bootstrap`, `hooks`.
10. **Given** `.gitignore`, **when** I create a `.env` file, **then**
    `git status` does not show it as untracked; **and** `uv.lock` /
    `pnpm-lock.yaml` are *not* ignored.
11. **Given** `docs/adr/0001-tooling-choices.md`, **when** I read it, **then** it
    records and justifies the `uv`, `pnpm`, and `just` choices.

## 8. Test plan

> All tests combined across the project must keep the suite under 2 minutes.
> This spec adds no application runtime, so its verification is near-instant.

- **Unit (pytest / Vitest):** none yet — pytest is not configured until spec 04.
  Do **not** add a test runner here.
- **Integration:** none. The only "integration" is the compose smoke check in
  acceptance criteria 3–4, executed manually / by the implementer (it requires
  Docker and so is *not* part of the automated fast suite).
- **E2E (Playwright):** not applicable at this stage.
- **Static verification (the real checks for this spec):**
  `uv sync --project backend`, `uv run ruff check backend`,
  `uv run ruff format --check backend`, `uv run mypy backend/src`,
  `pre-commit run --all-files`, and `just --list`. These run in well under a
  second and are the gate for "done".
- **Performance/budget note:** by design this spec contributes ~0s to the test
  budget. The compose healthcheck smoke test is excluded from the automated
  suite because it depends on Docker; spec 04 establishes the *automated* tiers
  using an ephemeral DB and fakes.
